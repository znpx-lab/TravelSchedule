# 統合テスト仕様書 (TEST.SPEC.md)

作成日: 2026-02-22
対象リポジトリ: TravelSchedule
基資料: spec.md

目的
- `spec.md` に定義された機能要件・非機能要件に基づき、システム全体の受入テスト仕様を作成する。
- 手動テストと自動化テストへの移行に必要な観点・テストケースを網羅する。

適用範囲
- フロントエンドの機能（スケジュール、場所、予算/費用、やること、メモ、設定）
- データ入出力（JSON エクスポート/インポート）
- 表示・操作性（レスポンシブ、アクセシビリティの基本）
- 主要ブラウザでの動作（Chrome / Firefox / Safari / Edge）

前提条件
- テスト実行マシンにブラウザがインストールされていること
- localStorage を利用するため、テスト前に現行データのバックアップを取得すること
- テストはローカルで `index.html` を直接開く方法で実行可能

環境一覧
- OS: Windows / macOS / Linux（最低1種で実施、クロスブラウザ確認）
- ブラウザ: Chrome 最新、Firefox 最新、Safari 最新（macOS）、Edge 最新
- 解像度: PC (>=1200px), Tablet (768-1199px), Mobile (<=767px)

テストスコープ（機能別）
1. 旅行（Trip）管理
2. スケジュール管理
3. 目的地・場所情報管理
4. 予算・費用管理（割勘 → 割勘表記）
5. やることリスト（タスク）
6. メモ・グローバルメモ
7. データエクスポート/インポート
8. UI/レスポンシブとアクセシビリティ
9. パフォーマンスとエラーハンドリング

各機能のテスト観点と代表ケース

1) 旅行（Trip）管理
- 観点:
  - 旅行の追加、編集、削除、切替が正しく動作する
  - currentTripId が正しく保存・復元される
- ケース:
  TC-TR-01: 旅行を新規作成 → 作成後ヘッダーのセレクタに表示される
  TC-TR-02: 旅行名を編集 → 変更が localStorage に反映される
  TC-TR-03: 旅行を削除 → currentTripId が存在しない場合に最初の旅行へフォールバック

2) スケジュール管理
- 観点:
  - イベント追加/編集/削除、時系列ソート、検索が機能する
  - イベントに場所を紐付けできる
- ケース:
  TC-SC-01: イベント追加（日時・タイトル）→ リストに表示
  TC-SC-02: イベント編集 → 変更が保存される
  TC-SC-03: 日付検索 → 指定日のイベントのみ表示
  TC-SC-04: イベントに場所を紐付け → 詳細にリンクが表示される

3) 目的地・場所情報管理
- 観点:
  - 住所、緯度経度、営業時間、画像、外部リンクの保存/表示
  - Google Maps 連携（リンク/埋め込み）が機能
- ケース:
  TC-LC-01: 場所登録（画像付き）→ 画像プレビューが表示される
  TC-LC-02: 緯度経度からMapリンクを生成 → クリックでGoogle Mapsを開く

4) 予算・費用管理（重点テスト）
- 観点:
  - 費用の追加/編集/削除
  - 合計金額表示の正確性
  - `割勘` 機能の計算ロジック（仕様に準拠）
  - `割勘対象のみ` トグルの挙動
  - `1人あたり` 表示の条件・強調表示
  - 端数表示の正しさ
  - UI表記が「割勘」に統一されていること
- 重点テストケース（抜粋）:
  TC-EX-01: 基本ケース (割勘 + 非対象混在)
    - 期待: 合計、割勘対象合計、割勘対象外合計、1人あたり、端数の表示が期待通り
  TC-EX-02: 端数検証（例: 25040 / 3 → perPerson=8346 remainder=2）
  TC-EX-03: 割勘人数=1 の非表示確認（1人あたり非表示）
  TC-EX-04: 割勘対象のみトグル ON の時に合計が割勘対象のみになること
  TC-EX-05: 金額未設定（null/未入力）は0として扱われること
  TC-EX-06: 大量データ(100+) の表示性能

5) やることリスト（タスク）
- 観点:
  - タスク追加/編集/完了/削除
  - 担当者割当、優先度、期限が反映されること
- ケース:
  TC-TK-01: タスク登録 → 一覧に表示、完了チェックで見た目が変化
  TC-TK-02: フィルタ（未完了/担当者）適用で期待の絞込み

6) メモ・グローバルメモ
- 観点: 保存、編集、エクスポート時に保持されること
- ケース: メモ編集→ JSON エクスポート → 再インポートで復元

7) データエクスポート/インポート
- 観点:
  - JSON エクスポートで localStorage の全データが出力される
  - JSON インポートでデータが復元される
  - バージョン差異やフォーマットの不整合時のハンドリング
- ケース:
  TC-IO-01: 現行データをエクスポート → ファイル内容が期待通り
  TC-IO-02: サンプルJSONをインポート → UI に反映
  TC-IO-03: 不正なフォーマットをインポート → エラー表示（もしくは拒否）

8) UI / レスポンシブ / アクセシビリティ
- 観点:
  - レイアウトが各ブレークポイントで崩れないこと
  - キーボード操作（タブ移動）で主要操作が可能であること
  - 目立たせる表現（1人あたり）が視覚的に十分であること
- ケース:
  TC-UI-01: 各解像度で主要画面を確認（スケジュール、予算、場所）
  TC-UI-02: キーボードで項目操作ができるか確認

9) パフォーマンス・エラーハンドリング
- 観点:
  - 大量データ時の表示・応答性能
  - 入力エラーや計算エラー時のユーザー向けメッセージ
- ケース:
  TC-PF-01: 200件のイベント/費用を読み込んでレンダリング時間を測定
  TC-PF-02: 電卓に不正式を入力 → 警告表示

テストデータ（サンプル）
- 既に作成済の簡易サンプルJSONをテスト開始時に利用すること。
- 追加サンプル: 大量データ用の自動生成スクリプト（簡易JSON生成）を用意すること推奨。

自動化の方針（任意）
- 単体計算ロジック（割勘算出）: ユニットテスト（Node.js + Jest 等）で自動化。
- E2E: Cypress や Playwright を用いて主要シナリオ（費用追加 → 割勘表示）を自動化。

合否基準
- 各 TC の期待結果が満たされること。
- 重要な表示崩れや計算差異が存在しないこと。
- 重大不具合がある場合は修正後に再テストを実施。

実行手順（推奨・簡易）
1. localStorage のバックアップ
2. サンプルJSONをインポート（または localStorage を直接書き換え）
3. 上記テストケースを順次実行し、スクリーンショットと結果を記録
4. 不具合は GitHub Issues に登録し、再現手順と期待値を添付

付録: 代表的な期待値表（抜粋）
- 入力: [1000(割勘), 500(非対象)], 割勘人数=2 → 合計=1500, 1人あたり=1000, 端数=0
- 入力: [25040(割勘のみ)], 割勘人数=3 → 合計=25040, 1人あたり=8346, 端数=2

担当とスケジュール
- テスト作成: 開発者（本コミット）
- 実行: QA チームまたは 開発者セルフチェック
- 実行期間（目安）: 1日〜2日（手動） / 自動化は別途スプリント計画

追記: 非機能閾値・JSONスキーマ検証・セキュリティテスト

1) 非機能要件の閾値（合否基準への明記）
- 初期ロード時間: 3秒以内（PC: Chrome 最新、ネットワーク条件: ローカル）
- 操作応答性: 500ms以内（主要ボタン押下からUI反映までの時間の目安）
- データサイズ: インポートする JSON は 1MB 以下で正常に処理できること

テストケース:
- TC-NF-01: 空の環境で `index.html` を開き、初回レンダリング時間を計測 → 3秒以内なら合格
- TC-NF-02: ボタン操作（費用追加、保存）を連続実行し、応答時間を計測 → 平均500ms以下を目標
- TC-NF-03: 1MB に近い JSON をインポートし、処理完了までの時間と成功を確認

2) JSON スキーマ検証（インポート時の入力検証）
- 目的: インポートされる JSON が仕様どおりの型・必須項目を満たしていることを確認する
- 必須フィールド例（抜粋）:
  - `trips[].id` (string), `trips[].name` (string), `trips[].expenses[].id` (string), `trips[].expenses[].amount` (number|null), `trips[].users[].id` (string), `currentTripId` (string)

テストケース:
- TC-SCH-01: 正常なサンプル JSON をインポート → UI に期待通り反映される
- TC-SCH-02: 必須フィールドを欠落させた JSON をインポート → インポート拒否または明確なエラーメッセージを表示
- TC-SCH-03: フィールド型が不正（文字列に数値など）の JSON をインポート → エラー表示
- TC-SCH-04: 日付フォーマットが不正（YYYY-MM-DD 以外） → 警告または自動修正の挙動確認

3) セキュリティ（簡易）テスト: XSS・不正入力対策
- 目的: ユーザー入力（タイトル、メモ等）によるスクリプト注入が行えないことを確認する

テストケース:
- TC-SEC-01: 費用タイトルやメモに `<script>alert(1)</script>` を入力し保存 → 表示時にスクリプトが実行されないこと
- TC-SEC-02: インポートする JSON に悪意あるスクリプトを埋め込む → インポート後に UI 表示時にスクリプトが実行されないこと
- TC-SEC-03: URL 表示フィールドに javascript: スキームを設定した場合に安全に扱われること（リンクとして直接開かれない等）

4) 自動化優先度（推奨）
- 高: 割勘計算ロジックのユニットテスト（期待値を含む多数ケース）
- 中: インポート/エクスポートのスキーマ検証（JSON ファイルの正誤判定）
- 中: 主要 E2E シナリオ（費用追加→合計・1人あたり表示）
- 低: レスポンシブのスクリーンショット比較（ビジュアル回帰）
